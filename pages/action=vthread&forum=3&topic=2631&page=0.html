почитал раздел [ Planning for the near future... ] файла whatsnew.txt и решил поделиться кое-какими идеями:

Интерфейс
---------
ComboBox со списком модулей больше в высоту, т.к. при наличии модулей шести (у меня во время отладки модулей их гораздо больше), приходится скролить (либо пихать '_' перед именем модуля).
На данный момент у combo Height=59, лучше сделать минимум 90

Алгоритм
--------
- сделать алгоритм генерации патча настраиваемым

Пример для ASPack v2.12
-----------------------
стандартный переход на OEP:

Code:
  1. push OEP
  2. ret


фактически выглядит так:
Code:
  1. 61            popad
  2. 75 08         jnz     @F
  3. B8 01000000   mov     eax, 1
  4. C2 0C00       ret     0C
  5. @@:
  6. 68 00000000   push    0
  7. C3            ret


как известно, push 0 преобразуется в push OEP, а
Code:
  1. mov eax, 1
  2. ret     0C

вызывается только в случае ошибки при распаковке
при этом до команды popad регистр EDX содержит ImageBase файла, а EAX - начало кода, что грех не использовать!
т.о. можно делать относительный патч
допустим, ImageBase файла 00400000, BaseOfCode - 1000, нам нужно пропатчить 00401015,
так вот, вместо
Code:
  1. mov      word ptr [00401015], 00EB ; = nop / null jmp

можно просто
Code:
  1. mov      word ptr [eax+15], 00EB

если смещение далеко от ImageBase (например, 00407015)
то сначала
Code:
  1. add eax, 6015

а потом уже
Code:
  1. mov      word ptr [eax], 00EB

т.о. экономится уйма места при пропатчивании большого кол-ва данных/кода
плюс ко всему можно делать переход на патч таким образом:
патч располагается по определённому смещению от начала файла, допустим, что в заголовке по адресу 7Ah
Благодаря ASPack, edx содержит ImageBase (400000), eax содержит ImageBase+BaseOfCode (401000)
патч лоадера аспака:
Code:
  1. add     edx, 7A
  2. call    near edx    ; 0040007A

код патча же выглядит так:
Code:
  1. [затёртая_только_что_команда] ; <- здесь будет команда, которую мы затёрли, выполнив переход на патч
  2. mov      byte ptr [eax+15], 0EB ; что-то патчим по адресу 00401015
  3. ret      ; возвращаемся на инструкцию после затёртой нами

при таком подходе значительно экономится место под патч, да и код для патча получается более универсальным
- возможность прикрутить поиск в памяти
на входе: стартовый адрес, размер данных для поиска, шаблон для поиска
на выходе: адрес начала данных из шаблона
в данном случае можно скомпилить код поисковика в памяти, а потом просто пихать его в тело пропатчиваемой проги, т.к. скомпиленый код будет "адресо-независимым"
пример:
Code:
  1.          mov     edx, TargetAddrStart
  2.          mov     eax, TargetSize
  3.          mov     ebx, Pattern
  4.          xor     ecx, ecx ; Counter
  5.          ; Let's find it!
  6.          Find_Sig_1:
  7.          cmp     dword ptr [edx+ecx], ebx
  8.          jne     Search_More_1
  9.          lea     eax, dword ptr [edx+ecx]
  10.          jmp Sign_1_Done
  11.          Search_More_1:
  12.                  inc    ecx
  13.                  cmp    ecx, eax
  14.                  jl     Find_Sig_1
  15.                  NoMatch:
  16.                  ret
  17.          Sign_1_Done:

- возможность использовать ассемблерные вставки, либо просто вставлять последовательность данных, например в HEX-виде
- возможность вручную задавать оффсет для патча
часть этого я пытался реализовать через модули, потому и пришлось их конвертить в масм, но всё равно нужного результата не смог добиться без поддержки со стороны хоста (собсно HzorInline'a)
